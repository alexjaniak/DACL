# Correctness Reviewer Memory

Persistent lessons and failure patterns from prior PR reviews.

- (init) Focus: behavior validation, tests, integration safety.
- When a PR introduces multiple entrypoint scripts for the same workflow, enforce a single canonical output contract (path + JSON schema). Drift here is a high-probability integration break.
- For bootstrap/setup tooling, dependency checks must run before any persistent side effects (branch/worktree/file creation). Fail-fast before mutation.
- Include explicit rerun/idempotency checks in reviewer evidence for agent bootstrap flows (same agent id should preserve wallet/pubkey unless rotation is intended).
- Include negative-path fail-fast evidence in review notes (prove missing deps do not create partial state).
- For repositories using git worktrees, validate metadata/worktree persistence against the git common root (not caller CWD/worktree), then re-check dependency fail-fast behavior after any path normalization change.
- 2026-03-04 reviewer loop: if PR/issue thread already contains iterative reviewer feedback, explicitly mark unresolved actionable items before proposing new critique; if none remain, post a verification summary with concrete command evidence and residual-risk statement instead of reopening settled threads.
- 2026-03-04 edge-case: `command -v` alone is insufficient for dependency preflight. Validate command functionality (`--version`/`version`) before mutation to catch shadowed or broken binaries that would otherwise fail mid-run after creating branches/worktrees.
